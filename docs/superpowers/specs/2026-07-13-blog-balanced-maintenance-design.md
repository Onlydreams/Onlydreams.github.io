# 个人技术博客平衡修复设计

## 背景

当前站点已经具备搜索、专题、索引、分类、标签、文章状态、目录、相关文章、RSS、评论和跨平台测试等完整能力。现阶段的主要问题不再是站点功能不足，而是首页与作者定位偏泛、分类标签过细、待复核文章比例较高，以及本机 Ruby 环境无法运行仓库锁定的 Bundler 版本。

本次修复以内容可信度和个人博客辨识度为中心，不继续扩展新的导航或站点功能。

## 目标

- 恢复当前机器运行完整仓库测试的能力。
- 明确博客聚焦 AI Agent 工具链、开发环境和网络排障的定位。
- 收敛分类与标签，减少只有单篇文章使用的过细 taxonomy。
- 逐篇联网核验当前 7 篇“待复核”文章，并按证据更新状态、正文和更新时间。
- 保持现有文章 URL、站名、域名和整体视觉风格不变。

## 非目标

- 不增加新的导航页面、评论系统、统计平台或第三方依赖。
- 不更换 Jekyll、Minima、Ruby 或 Bundler 的仓库锁定版本。
- 不对所有文章做风格重写，也不为了扩大篇幅添加无关内容。
- 不把截图数量作为验收指标；无法可靠复现的历史界面不补模拟截图。
- 不修改文章 permalink，不破坏现有搜索收录和外部链接。

## 设计方案

### 1. 构建环境

仓库继续以 `.ruby-version` 和 `Gemfile.lock` 为唯一版本来源。当前机器使用 Ruby 3.3.11，但该 Ruby 旁缺少 Bundler 2.5.22，因此完整 `bash bin/test` 无法启动。

修复只为当前 Ruby 3.3.11 安装仓库要求的 Bundler 2.5.22，不设置持久的 `BUNDLE_PATH`、`GEM_HOME`、`GEM_PATH` 或 `RUBYOPT`，也不修改 lockfile。安装若需要写入系统 Ruby 目录，执行前单独请求权限。

### 2. 首页与 About 定位

首页首屏使用具体价值主张替代泛化欢迎语，明确内容范围为：

- AI Agent、Skills 与协作工作流；
- macOS、Windows 等开发环境和工具链；
- 网络、代理与真实故障排查；
- 带验证日期、适用环境和风险说明的可复现技术笔记。

首页增加轻量的“从这里开始”入口，链接到现有代表性文章或专题，不新增独立页面。专题卡片和最新文章列表保持现有结构。

About 页面补充：

- 作者关注方向；
- 写作原则，包括可复现、标注环境、公开风险和持续复核；
- “当前可用 / 待复核 / 已失效”的含义和文章状态入口；
- `Onlydreams` 站名与 `dayjia.com` 域名同属该个人博客的简短说明；
- 现有 GitHub、文章评论联系方式。

### 3. Taxonomy 收敛

分类承担稳定主题导航职责，收敛为明确白名单：

- `AI`
- `开发工具`
- `网络与代理`
- `浏览器`
- `体育技术`

现有过细分类映射如下：

- `Agent`、`工具`、`配置`、`效率` 合并到文章最相关的稳定分类，不机械地为每篇文章保留多个大类。
- `MacOS`、`Homebrew`、`Git`、`GitHub` 作为具体平台或工具下沉到标签。
- `代理` 合并为 `网络与代理`。
- `Sports` 合并为 `体育技术`。

每篇文章保留一到两个分类。标签继续承担工具、平台和问题关键词职责，通常保留四到六个；合并明显同义变体，但允许具体工具名只出现一次，因为它们仍有搜索价值。测试只禁止已经明确列出的同义变体，不设置“标签必须至少出现两次”这类会损失检索能力的硬规则。

### 4. 待复核文章核验

对以下 7 篇文章逐篇核验：

- `2026-04-25-macos-homebrew-acceleration.md`
- `2026-04-30-macos-claude-deepseek.md`
- `2026-05-04-auto-proxy-setup.md`
- `2026-05-04-skillshare-guide.md`
- `2026-05-17-codex-desktop-gpu-rendering-bug.md`
- `2026-05-26-clash-verge-github-node-speed-test.md`
- `2026-07-12-codex-app-superpowers-plugin-missing.md`

每篇文章提取产品版本、命令、下载入口、配置路径和核心结论。证据优先级为：官方文档、官方仓库、官方 issue 或 release；官方资料不足时才使用可信第三方来源。

核验结果按以下规则处理：

- **仍然有效**：状态改为或保持 `当前可用`，`verified` 使用 `2026-07-13`，必要时补充当前验证环境。
- **部分变化**：保留仍有效内容，修正已经变化的命令、入口或限制，并在正文说明适用边界。
- **已失效但仍有历史价值**：状态改为 `已失效`，正文开头说明失效原因和当前替代路径。
- **无法可靠确认**：保持 `待复核`，明确未确认项，不根据推测填入结论。

只有正文发生实质变化时才添加或更新 `updated: 2026-07-13`。单纯更新 `status.verified` 不视为正文实质更新。

公开内容继续脱敏真实密钥、Token、个人路径、代理节点、控制接口和内部环境信息。第三方工具使用中性描述，不加入导流或推广文案。

### 5. 图片与视觉证据

截图不是本轮硬性范围。只有同时满足以下条件时才添加：

- 能在当前环境可靠复现；
- 图片能证明关键现象或验证结果；
- 不包含个人账号、路径、节点、Token 或其他敏感信息。

否则使用版本表、命令输出示例或文字边界说明替代，不生成模拟截图。

## 测试设计

站点行为测试新增以下约束：

- 首页包含新的具体定位文案和“从这里开始”入口。
- 精选入口只链接现有文章或专题。
- About 页面包含关注方向、写作原则、状态说明和站名/域名关系。

内容健康测试新增以下约束：

- 所有文章分类必须来自分类白名单。
- 每篇文章只能包含一到两个分类。
- 标签不得包含设计中明确合并的同义变体。
- `当前可用` 仍必须使用 ISO 格式验证日期。

实施遵循红绿循环：先写测试并确认因目标行为缺失而失败，再修改页面或文章使其通过。

## 验收标准

- 7 篇待复核文章都有明确、可追溯的核验结论。
- 发现变化的正文已修正，并正确更新 `updated`；没有变化的正文不制造更新时间噪声。
- 首页和 About 能清楚说明博客的主题、写作原则和可信度机制。
- 所有文章分类均来自五个稳定分类，每篇一到两个分类。
- 文章 URL、站名、域名和现有核心导航保持不变。
- `bash bin/test` 完整通过。
- `git diff --check` 通过。
- 最终敏感信息检查通过，工作树只包含本次目标改动。

## 风险与处理

- 外部产品可能在核验期间继续变化：最终结论记录核验日期，并避免把短期界面细节写成永久规则。
- 某些历史故障无法在当前硬件或操作系统复现：保留历史价值，明确证据边界，不声称重新实测。
- taxonomy 调整会改变分类页结构：不改变文章 URL，并通过站点构建测试确认分类入口仍可访问。
- 本机 Bundler 修复涉及仓库外安装：只安装 lockfile 指定版本，不修改持久环境变量。
